# Basic Authentication Design

Date: 2026-05-22

## Context

CAM is a small FastAPI app deployed on Vercel. It serves a static internal UI and exposes Canto-backed API routes from the same app. The current deployment has no application-level access control, so anyone who can reach the deployed URL can attempt to load project metadata and trigger Canto updates.

The goal is to add very basic authentication without paid services, external identity providers, Supabase, Vercel authentication products, or a database.

## Decision

Use a single shared password with a signed HTTP-only session cookie.

The password and cookie signing secret will be configured with environment variables:

- `CAM_AUTH_PASSWORD`: shared password for the tool.
- `CAM_SESSION_SECRET`: long random secret used to sign session cookies.

This is intentionally simple. It protects the internal tool and API routes, works on Vercel serverless functions, and avoids user management, password reset flows, and persistent storage.

## Non-Goals

- No individual user accounts.
- No email-based password reset.
- No database.
- No OAuth or third-party authentication service.
- No role-based permissions.
- No audit log of individual users.

## User Flow

1. An unauthenticated user visits `/`.
2. The app redirects the user to `/login`.
3. The user enters the shared password.
4. If the password is correct, the backend sets a signed HTTP-only cookie and redirects to `/`.
5. The user can use the existing CAM interface and API routes normally.
6. The user can visit `/logout` to clear the session cookie.

If the password is forgotten, an operator changes `CAM_AUTH_PASSWORD` in Vercel and redeploys. If all sessions must be invalidated, the operator rotates `CAM_SESSION_SECRET`.

## Route Protection

Authentication middleware will run before normal route handling.

Allowed without a session:

- `GET /login`
- `POST /login`
- Static assets needed by the login page, such as `/static/colors_and_type.css`, fonts, and the Fairpicture logo

Protected:

- `/`
- `/docs`, `/redoc`, and `/openapi.json`
- `/api/projects`
- `/api/matches/{project_id}`
- `/api/decision`
- `/api/decision/bulk-confirm`
- Any future `/api/*` route unless deliberately exempted

For browser page requests, unauthenticated access redirects to `/login`. For API requests, unauthenticated access returns `401` JSON instead of an HTML redirect.

## Components

### `auth.py`

A small auth module will own session behavior:

- Read `CAM_AUTH_PASSWORD` and `CAM_SESSION_SECRET`.
- Validate submitted passwords with constant-time comparison.
- Create a signed session token.
- Verify a signed session token.
- Build cookie parameters consistently.

The module should use `itsdangerous` for signed cookies rather than custom HMAC code. Add `itsdangerous` to `requirements.txt`.

### `cam_app.py`

The FastAPI app will:

- Register auth middleware.
- Add `GET /login`.
- Add `POST /login`.
- Add `GET /logout`.
- Keep existing API and static routes unchanged except for authentication checks.

The login response can be a compact HTML page generated by Python or a static file. It should reuse the existing Fairpicture CSS and logo where practical, but the first implementation should stay minimal.

## Cookie Behavior

The session cookie should be:

- HTTP-only.
- SameSite `lax`.
- Secure when running on Vercel or when `CAM_AUTH_COOKIE_SECURE=true`.
- Path `/`.
- Expiring after a fixed duration, initially 7 days.

The cookie should not store the password. It should store only a signed session marker, such as `authenticated`.

## Error Handling

If auth environment variables are missing:

- In production, protected requests should fail closed with a clear server-side error.
- Locally, the startup/import path should remain usable enough for tests, but login should not succeed unless a password and secret are configured.

If login fails:

- Return the login form with a short invalid-password message.
- Do not reveal configuration details.

If a cookie is missing, expired, malformed, or signed with an old secret:

- Treat the request as unauthenticated.

## Security Notes

This design is basic protection for an internal operational tool. It is not a full identity system.

Primary protections:

- The password is stored in Vercel env vars, not in git.
- The cookie is signed and HTTP-only.
- API routes are protected as well as the UI.
- Rotating `CAM_SESSION_SECRET` invalidates existing sessions.

Known limitations:

- Everyone shares one credential.
- There is no per-user audit trail.
- Anyone with the shared password has full CAM access.
- Password rotation is manual.

## Testing

Add focused tests around the authentication boundary:

- `auth.py` signs and verifies a session token.
- Invalid or expired tokens fail verification.
- Wrong password is rejected.
- Correct password is accepted.
- `GET /` redirects to `/login` without a session.
- `GET /api/projects` returns `401` without a session.
- `POST /login` with the correct password sets a session cookie.
- Authenticated requests can reach protected routes.
- `GET /logout` clears the session cookie.

Tests that touch Canto-backed routes should mock the Canto client to avoid network calls.

## Deployment

Set these Vercel environment variables before or with deployment:

- `CAM_AUTH_PASSWORD`
- `CAM_SESSION_SECRET`

Generate `CAM_SESSION_SECRET` as a long random string. A password manager or `openssl rand -base64 32` is sufficient.

After pushing the implementation, Vercel should redeploy automatically from `main`.
